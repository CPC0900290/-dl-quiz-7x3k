Q1. 團隊的 ViewModel 用 protocol ViewControllerDelegate，讓 ViewController 遵從並設為 viewModel.delegate，ViewModel 內部資料變化時呼叫 delegate?.didUpdateUser(_:) 通知畫面更新。有人說「這其實不是 MVVM」，這個說法的依據是什麼？
解析

正解 B。MVVM 的核心契約是「View 認識 ViewModel，但 ViewModel 對 View 一無所知」——資料流是單向的觀察關係，ViewModel 只管把狀態放出去，不在乎誰在看、看的人要做什麼。用 delegate 綁定時，ViewModel 必須先定義一份 protocol（例如 didUpdateUser(_:)），這份 protocol 描述的正是「View 該有什麼行為」；ViewModel 指定了誰是它的 delegate，等於它明確知道自己在跟一個特定角色的物件溝通、並主動呼叫對方的方法——這已經是 MVP 的責任邊界（Presenter 主動命令 View 做事），不是 MVVM 的被動資料流。用 weak var delegate 只解決 retain cycle，不影響「ViewModel 是否認識 View」這個架構歸屬問題。

相對地，用 closure 綁定（viewModel.onUserUpdated = { user in ... }）時，ViewModel 只是把一個值透過 closure 丟出去，它不知道這個 closure 裡面做了什麼、也不知道是誰設的——這保留了 MVVM 該有的「View 不透明」特性。這個對照可以參考你自己筆記 [[MVVM]] 裡對 closure vs delegate 綁定方式的整理，裡面也記了這篇討論的原始出處。

面試官可能追問：「那 RxSwift 的 Observable<User> 綁定算哪一種？」— 算 MVVM，甚至更純粹：ViewModel 只 expose 一個 stream，完全不知道有沒有人訂閱、訂閱的人是誰、訂閱後做了什麼，資料流方向與知情範圍都比 closure 更嚴格地單向。
架構設計 · 方案 trade-off
Q2. 你的 App 目前用 MVVM，Model 層直接讓 ViewModel import 並呼叫 URLSession 發 API request。主管建議導入 Clean Architecture 的 UseCase + Repository 分層。在一個 3 人團隊、功能仍快速迭代中的產品，這個建議最主要的價值該怎麼講？
解析

正解 B。Clean Architecture 分層的價值不是「程式碼變少」或「效能變好」，而是依賴方向的控制：讓 ViewModel（or Presenter）依賴一個抽象的 UserRepository protocol，而不是具體的 URLSession。這樣做測試時可以直接注入 mock repository，不需要啟動真實網路請求；未來如果資料來源改變（加上 local cache、換成 GraphQL、offline-first），只需要改 Repository 的實作，UseCase 和 ViewModel 完全不用動。這正是你在 [[Dependency Injection (DI)]] 筆記裡記錄的 protocol-based DI 精神——把「誰提供這個能力」和「怎麼使用這個能力」解耦。

C 選項是常見的過度推論陷阱：架構決策沒有「業界最佳實踐、無條件套用」這種答案，永遠是 trade-off——3 人團隊快速迭代期，多一層 UseCase/Repository 意味著新增功能要多寫幾個檔案、多一層心智負擔，如果 ViewModel 目前只有 1-2 個資料來源且沒有測試需求，過度分層反而拖慢迭代速度。真正該問的是：「這個產品現在或可預期的未來，是否有『需要替換資料來源』或『需要單元測試覆蓋商業邏輯』的實際需求？」如果有，分層的投資才划算。

面試官可能追問：「如果現在專案還小，你會直接導入完整 Clean Architecture 嗎？」— 更務實的答案是漸進式：先把「網路呼叫」抽成 protocol（Repository 雛形），確保 ViewModel 不直接 import URLSession；等真的出現多資料來源或測試覆蓋的需求時，再補上 UseCase 層。避免一開始就過度設計。
隨機主題：RxSwift · 實務情境判斷
Q3. 你的 ViewModel 有一個 let userTapped = PublishSubject<Void>() 當作 Input，多個 UI 元件（button 的 rx.tap）都綁定到它。某次重構後，某個 button 的 rx.tap.bind(to: userTapped).disposed(by: bag) 少接了 disposed(by:)，結果什麼問題最可能發生？
解析

正解 B。沒接 disposed(by:) 不代表立即 crash，也不是 Subject 自動清除——訂閱本身建立的 closure 會被存進 PublishSubject 內部的 observers bag（一份記錄所有訂閱者的清單），這是「被全域/長生命週期物件延長生命週期的單向強引用鏈」，跟兩個物件互相持有的 retain cycle（封閉互相持有）是不同機制：這裡沒有「互相」持有，只有 Subject 單向地一直記著這個訂閱，只要 Subject 還活著，訂閱者的 closure 就不會被釋放。

如果這個 closure 捕捉了 self（VC 或某個 view），即便 VC 已經從畫面移除，只要 ViewModel（進而 Subject）還活著，這個 closure 對 self 的強引用鏈就還在，導致 VC 無法被釋放——這是記憶體洩漏，但成因跟「A 持有 B、B 持有 A」的封閉環不同，解析與修法不能混用這兩種機制的說法。這題呼應你在 [[RxSwift]] 筆記中對 DisposeBag 管理生命週期的重點：每個訂閱都該綁定到一個明確的生命週期範圍（VC 的 bag 或 cell 重用時重建的 bag），不接 disposed(by:) 幾乎永遠是疏忽而非設計選擇。

面試官可能追問：「如果是 cell 裡的 rx.tap 綁定，忘記 disposed 有什麼特別的風險？」— cell 重用時如果沒有在 prepareForReuse 重建 DisposeBag，舊的訂閱會持續疊加，同一個 tap 事件可能觸發多次過期的 handler，這是 UITableView/UICollectionView cell 重用場景下最常見的 Rx 陷阱。
🌐 後端必備知識
1-6 GraphQL 與 gRPC 概觀：與 REST 的取捨

前面五節都在談 REST，這節把視野拉開：REST 不是唯一解法。GraphQL 和 gRPC 都是為了解決 REST 在特定場景下的限制而生的，理解它們的差異，最終還是為了回答同一個問題——「這個 API 該怎麼設計？」

REST 的兩個典型痛點

想像一個「貼文列表」畫面，你的 App 需要作者名字、貼文標題、前 3 則留言。用 REST，如果 GET /posts 只回傳貼文本體，你還得再打 GET /users/:id 和 GET /posts/:id/comments——一個畫面打 3 個 API，這是 under-fetching。反過來，如果 GET /posts 把作者的 30 個欄位全部塞進 response，但畫面只需要名字，這是 over-fetching——行動網路下，多餘的 bytes 都是真實的流量與電量成本。

GraphQL：把「決定回傳什麼」的權力交給 client

GraphQL 用一份 schema（型別系統，類似 Swift 的 protocol/struct 定義）描述所有可查詢的資料形狀，client 端寫 query 精確指定要哪些欄位，server 端每個欄位背後有一個 resolver 負責取值。這解決了 over/under-fetching，但引入了新問題：一個 list 裡每個 item 都去查一次關聯資料（例如每篇貼文都查一次作者），會變成 N+1 查詢，需要用 DataLoader 把同一輪次的查詢合併成一次批次查詢。這部分的完整 schema 範例、resolver 程式碼、DataLoader 原理，已經整理在你自己的筆記 [[GraphQL 深度解析]] 和 [[GraphQL on iOS]] 裡，包含 Apollo iOS 2.0 的 async/await 用法與 normalized cache 設計。

gRPC：把「格式」和「傳輸」都最佳化

gRPC 走的是完全不同的路線：不追求查詢彈性，而是用 Protocol Buffers（二進位序列化格式）取代 JSON，並以 function call 的思維取代 REST 的 resource 思維——呼叫 getUser(id:) 而不是 GET /users/:id。二進位格式體積比 JSON 小、不需要 string parsing；搭配 HTTP/2 原生支援，還能做 4 種 streaming 模式（unary / server streaming / client streaming / bidirectional streaming），特別適合聊天、即時協作這類場景。

你之前已經寫過 gRPC Swift 2（全面 async/await + AsyncSequence）的完整筆記 [[gRPC on iOS]]，裡面涵蓋 Protobuf 的 field number 設計、四種 streaming 模式的程式碼範例、以及 Apple 在 WWDC26 推廣 gRPC Swift 的脈絡，值得對照複習。

三者怎麼選：一張表看懂 trade-off

REST 學習成本最低、生態最成熟，適合公開 API 和簡單 CRUD；GraphQL 適合資料形狀多變、多種 UI 消費同一份資料的場景（BFF 層常見選擇）；gRPC 適合你自己控制前後端、追求效能與原生 streaming 的內部服務通訊。三者不是互斥——很多系統是「對外用 REST、內部微服務間用 gRPC、給前端的 BFF 層用 GraphQL 聚合」。

小測驗

B1. GraphQL 解決的「over-fetching」和「under-fetching」分別指什麼？
解析

B 正確。Over-fetching：resource 回傳整包但你只用到幾個欄位（浪費頻寬）；under-fetching：一個 resource 拿不到全部你要的資料，要多打幾支 API 補齊（浪費往返次數）。GraphQL 讓 client 一次 query 精確指定欄位，同時解決兩者。

B2. GraphQL 的 N+1 問題最根本的成因是什麼？
解析

B 正確。resolver 是逐欄位執行的，如果 Post.author resolver 沒有做批次處理，10 篇貼文就觸發 10 次獨立查詢。DataLoader 把同一個 event loop tick 內的多次查詢請求合併成一次 WHERE id IN (...)，解決這個問題。

B3. 為什麼 gRPC 特別適合聊天室這類即時雙向通訊的場景，而不是靠 REST + 輪詢？
解析

B 正確。REST 一問一答的模型要做「即時」只能靠輪詢（浪費資源）或另外接 WebSocket；gRPC 的四種 streaming 模式（尤其 bidirectional streaming）是協議原生支援的，同一條 HTTP/2 連線雙方隨時互推資料，天然適合聊天、協作、遊戲。

🧩 演算法
題型輪替：Linked List（第一次出現 → Easy）
LeetCode 206. Reverse Linked List
給定一個單向鏈結串列的 head node，請將整個串列反轉，回傳反轉後的 head。 

輸入: head = [1,2,3,4,5]
輸出: [5,4,3,2,1]

輸入: head = [1,2]
輸出: [2,1]

輸入: head = []
輸出: []
進階要求：先想出迭代解（O(1) 額外空間），再想遞迴解（O(n) call stack）。
提示 1：思考方向

Linked list 反轉的本質是：把每個 node 的 next 指標方向「倒過來」。原本 1 → 2 → 3，反轉後要變成 1 ← 2 ← 3，也就是 3 → 2 → 1。難點在於：一旦你把某個 node 的 next 改指向前一個 node，你就失去了原本「往後走」的路——所以動手改之前，得先想清楚要保留哪些暫存指標，才不會斷鏈。

提示 2：關鍵觀察

用三個指標：prev（一開始是 nil，代表反轉後新串列的結尾）、curr（目前處理到的 node，一開始是 head）、next（暫存 curr.next，因為等一下要覆寫它）。每一輪迴圈做四件事：先存 next = curr.next（保命索），再把 curr.next = prev（真正的反轉動作），然後 prev = curr、curr = next（整體往前推進一格）。迴圈跑到 curr == nil 結束，此時 prev 就是新的 head。這跟你在 UIKit 裡处理 doubly-linked 結構（例如手刻 LRU Cache）時「動指標前一定要先存下一步要用的引用」的直覺是同一套原則。

提示 3：參考解法（Swift） 
// Definition for singly-linked list node
class ListNode {
    var val: Int
    var next: ListNode?
    init(_ val: Int) {
        self.val = val
        self.next = nil
    }
}

// 迭代解：O(n) 時間、O(1) 額外空間
func reverseList(_ head: ListNode?) -> ListNode? {
    var prev: ListNode? = nil
    var curr = head

    while curr != nil {
        let next = curr?.next
        curr?.next = prev
        prev = curr
        curr = next
    }
    return prev
}

// 遞迴解：O(n) 時間、O(n) call stack 空間
func reverseListRecursive(_ head: ListNode?) -> ListNode? {
    guard let head = head, let next = head.next else {
        return head  // 空串列或只剩一個 node，本身就是反轉後的 head
    }
    let newHead = reverseListRecursive(next)
    next.next = head   // 讓下一個 node 反過來指回自己
    head.next = nil     // 自己變成新的尾巴，切斷原本往後的指標
    return newHead
}

複雜度：迭代解時間 O(n)、空間 O(1)；遞迴解時間 O(n)、空間 O(n)（call stack）。ListNode 是 class（reference type），這題也是觀察「多個變數持有同一個 heap 物件的引用」的好練習——prev、curr、next 三個變數在反轉過程中互相傳遞的是引用而非複製，這跟 Swift 的 struct/class 記憶體行為直接相關，可對照 [[Value Type vs Reference Type 的記憶體行為]]。這是 linked list 題型的基礎題，下次輪到時難度會升到 Medium（例如 Merge Two Sorted Lists 或 Remove Nth Node From End），三指標操作的手感要練到位。
