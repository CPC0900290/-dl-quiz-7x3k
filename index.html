2 題 UIKit + 1 題隨機（記憶體管理）
Q1. 你的 List 頁面用 self-sizing cell（UITableView.automaticDimension），Cell 內部大概有 25 個 constraints。上線後某些機型滑動時偶爾掉幀，Instruments 顯示大量時間花在 systemLayoutSizeFitting。下列哪個說法最準確地解釋了問題根源？ 
tableView.rowHeight = UITableView.automaticDimension
tableView.estimatedRowHeight = 44   // Cell 實際平均高度是 180

這是面試官很愛問的「為什麼」而不是「怎麼修」：self-sizing 的本質是系統在 cellForRow 之後呼叫 cell.systemLayoutSizeFitting(_:)，讓 Auto Layout 引擎實際求解一次 constraint system 才能得到 intrinsic content size。這個求解成本隨 constraint 數量增加而快速上升，25 個 constraints 屬於偏重的量級。同時 estimatedRowHeight 只是一個「預估值」，如果跟實際高度差太多（這裡差了 4 倍），滾動過程中系統要不斷用估計值排版、拿到真實高度後又重新調整，造成 scroll indicator 跳動與額外的 layout pass——這兩件事是分開但會疊加的成本，不是同一個問題的兩種說法。

相關筆記：[[UITableView-UICollectionView 效能優化]]

面試官可能追問：「如果不能減少 constraint 數量，還有什麼優化方向？」→ 手動快取已計算過的高度（[IndexPath: CGFloat]）、把 estimatedRowHeight 設成接近實際平均值、或改用 sizeThatFits(_:) 手動計算取代讓 Auto Layout 引擎求解。
Q2. 設計師要求卡片有陰影效果，你這樣寫，結果在 Instruments 的 Core Animation 工具裡看到大片黃色（Color Offscreen-Rendered）： 
layer.shadowColor = UIColor.black.cgColor
layer.shadowOffset = CGSize(width: 0, height: 2)
layer.shadowRadius = 4
layer.shadowOpacity = 0.3
// 沒有設定 shadowPath
下列哪個修法最直接對症下藥，且說明了正確的原因？

Shadow 沒有指定 shadowPath 時，GPU 沒辦法「猜」出 layer 的輪廓要怎麼畫陰影，只能先把整個 layer tree 完整渲染到一個暫時的 offscreen buffer，再從裡面推算出 alpha 輪廓，這一整套流程包含額外的記憶體分配、context switching、以及多一次完整的 rendering pass。加上 shadowPath 之後，GPU 直接照著你給的路徑畫陰影，完全跳過那次額外渲染。要注意如果卡片的 bounds 會隨 Auto Layout 變動（例如 self-sizing cell），shadowPath 必須在 layoutSubviews 裡跟著更新，否則陰影形狀會跟畫面對不上。

相關筆記：[[離屏渲染與 Core Animation 效能]]

面試官可能追問：「那 cornerRadius 加 masksToBounds 呢，一定會觸發 offscreen rendering 嗎？」→ 不一定。iOS 13+ 對「只裁切 backgroundColor/borderWidth，沒有子 view 需要裁切」的情況做了優化不再觸發；只有當 layer 有子 view 或 contents 需要被裁切、且設了 masksToBounds 時才會觸發。
Q3.（隨機主題：記憶體管理）以下 ViewModel 只在 viewDidLoad 訂閱了 App 層級 singleton 的 UserSession.shared.stateRelay（不是自己的 property），畫面 pop 離開後 deinit 沒有被呼叫。disposeBag 有正確在 deinit 前被釋放。下列哪個說法正確描述了這個現象的機制？ 
class ProfileViewModel {
    let disposeBag = DisposeBag()

    func bind() {
        UserSession.shared.stateRelay
            .subscribe(onNext: { [weak self] state in
                self?.apply(state)
            })
            .disposed(by: disposeBag)
    }
    deinit { print("🗑️ deinit") }
}

這題刻意反著問，用來檢查「是否真的理解機制，還是背了『訂閱 singleton 會漏』這種片面結論」。正確的判讀框架是分清楚三種不同的「物件死不掉」：真環（retain cycle）是封閉的互相持有（例如訂閱自己的 Relay 又強捕獲 self）；假環（全域根強鏈）是被活得比你久的全域物件單向持有（例如訂閱 singleton 的 subject、repeating Timer 的 target、NotificationCenter observer），這種情況下即使沒有互相持有，只要 closure 強捕獲了 self，一樣會讓 self 活到跟 singleton 一樣久；而題目給的程式碼因為已經正確用了 [weak self]，closure 根本沒有強捕獲 self，不管訂閱的是自己的 Relay 還是 singleton 的 Relay，這條路徑都不構成阻止 deinit 的原因。把「訂閱 singleton 一定漏」當成規則來背，會讓你在這種已經正確處理過的程式碼裡誤判出根本不存在的 bug。

相關筆記：[[引用鏈判讀：真環、假環與 struct 穿透]]、[[Retain Cycle 完整圖鑑]]

面試官可能追問：「如果拿掉 [weak self]，訂閱 singleton 的 Relay 會發生什麼事？」→ 會變成假環：UserSession.shared → stateRelay → observers Bag → closure → self，因為 UserSession.shared 是 App 級全域根，這條單向強鏈會讓 ViewModel 活到跟整個 App 一樣久，即使它不是封閉循環，也不會被 ARC 自動偵測或回收。

🌐 後端必備知識 1-3 TLS 與 HTTPS

Stage 1：HTTP 與 Web 基礎

昨天談完 HTTP 怎麼「送資料更快」，今天談送資料「怎麼安全」。HTTPS 就是 HTTP + TLS——TLS 在 TCP 之上加了三個保證：機密性（第三方看不到內容）、完整性（內容沒被竄改）、身份驗證（你連的真的是那台 server，不是冒充的）。這對 App 工程師來說不只是「後端的事」：你在 URLSession 裡處理的 URLAuthenticationChallenge、Certificate Pinning 的實作，都是直接站在這套機制上面。

先看一個具體場景：App 打開時第一次連線

假設你的 App 冷啟動後第一次呼叫 GET /api/profile。在 TCP 三次握手完成之後，TLS 握手才開始——這是額外疊加的一層，每多一輪來回（RTT）都是實打實的延遲，在行動網路上尤其有感。TLS 1.2 需要 2-RTT 才能開始傳真正的資料：

TLS 1.2：2-RTT 才能開始傳輸應用層資料

關鍵瓶頸在於 TLS 1.2 的握手是「一來一往、逐步確認」：Client 先問 server 支援什麼，server 回覆選定的版本與 certificate，Client 驗證完才能送出 key exchange 資料，server 再確認一次——完整跑完要兩輪。TLS 1.3 把這個流程重新設計，讓 Client 在 第一則訊息就直接帶上自己的 DH public key（key_share），不用等 server 先選好演算法再送，因此壓縮成 1-RTT，甚至對「回頭客」（曾經連過的 session）可以做到 0-RTT：

TLS 1.3：1-RTT，且從 EncryptedExtensions 開始整段都是加密的

除了少一輪 RTT，TLS 1.3 還把 Certificate 這類敏感資訊改成加密傳輸（TLS 1.2 是明文送出 certificate chain，任何中間節點都看得到你連的是哪個網域），並且移除了一批已知不安全的演算法（RSA key exchange、CBC 模式、SHA-1），強制要求 Forward Secrecy——每次連線用臨時產生的 ephemeral key，即使 server 的長期 private key 未來洩漏，過去被側錄的流量也無法被回頭解密。

Certificate Chain：為什麼 App 要信任一份「陌生」的憑證

Server 送來的 certificate 不是憑空可信的，而是靠一條 信任鏈 逐層驗證回到系統內建的 Root CA：

Root CA（預裝在 iOS 系統，Apple 管理）
   └─ Intermediate CA（Root CA 簽發）
        └─ Server Certificate（api.yourapp.com，Intermediate CA 簽發）

驗證時要逐層確認 signature、有效期、網域是否匹配，直到鏈頂的 Root CA 出現在系統信任清單裡才算過。這也是為什麼「開發環境用自簽 certificate 導致連線失敗」——自簽憑證沒有任何 CA 簽發，鏈直接斷在半路，系統當然不信任。

那 Certificate Pinning 又是為了防什麼？

標準 TLS 驗證回答的問題是「這張憑證是被信任的 CA 簽的嗎？」，但如果使用者手機被裝了惡意 root CA（常見於某些 MDM 或惡意軟體），或是某個 CA 本身被入侵、簽發了假憑證，攻擊者一樣能提供一張「合法但假」的憑證通過標準驗證，做中間人攻擊（MITM）。Certificate Pinning 回答的是另一個更嚴格的問題：「這張憑證是不是我事先預期的那一張／那把公鑰？」——即使是合法 CA 簽的其他憑證也一律拒絕。這是防禦層級的差異，不是同一件事的兩種說法。

iOS 端的落地：ATS 與 URLSessionTaskMetrics

iOS 9 起預設啟用 ATS（App Transport Security），強制要求 HTTPS、TLS 1.2 以上、Forward Secrecy、以及最低金鑰長度，等於把上面這些安全底線直接內建在系統層級，你不需要每個 App 都自己檢查一遍。如果想確認實際連線用的是哪個 TLS 版本，可以在 didFinishCollecting metrics: 裡讀 transaction.negotiatedTLSProtocolVersion；想觀察握手花了多久，可以比較 secureConnectionStartDate 和 secureConnectionEndDate 的差值——這在 debug「App 冷啟動第一個 API 特別慢」時很有用，往往就是 TLS 握手加上 DNS 查詢疊加起來的延遲。

📝 小測驗（3 題，點擊展開作答）
B1. TLS 1.3 相較 TLS 1.2，握手 RTT 數量減少的關鍵原因是什麼？

TLS 1.2 之所以要 2-RTT，是因為 Client 要先知道 Server 選定的 cipher suite 之後，才能送出對應的 key exchange 參數。TLS 1.3 讓 Client 主動先猜測（實務上幾乎都猜對，因為支援的曲線很集中）並直接把 key_share 塞進第一則訊息，Server 收到後可以立刻回覆自己的 key_share 加上已經加密的 Certificate，完成 1-RTT。Certificate 驗證流程仍然存在，只是被加密保護且塞進同一輪 RTT 內。UDP 傳輸是 HTTP/3 的 QUIC 層面的事，跟 TLS 1.3 是否用 TCP 無關（TLS 1.3 一樣可以跑在 TCP 之上）。

B2. 什麼是 Forward Secrecy？它保護的是哪種情境？

如果 key exchange 用的是固定的 RSA 長期金鑰，一旦這把 private key 未來洩漏，攻擊者可以拿它回頭解密所有曾經側錄下來的加密流量（因為每次連線用的都是同一把金鑰衍生的 pre-master secret）。Forward Secrecy 靠每次連線都產生一組全新、用完即丟的 ephemeral key pair（ECDHE 的「E」），讓每次連線的金鑰互相獨立，過去的流量不會因為未來金鑰外洩而被回溯破解。這跟封包遺失重傳、certificate 效期、惡意 root CA 都是不同層面的問題。

B3. Certificate Pinning 和標準的 TLS Certificate 驗證，兩者的差異是什麼？

標準 TLS 驗證的信任基礎是「這張證書的簽發鏈可以回溯到系統信任的 Root CA」，但這個信任模型有個前提：所有 CA 都值得信任、都不會被騙或被駭。Certificate Pinning 不依賴這個前提，直接把「我只信任這一張特定憑證（或其 public key）」寫死在 App 裡，即使攻擊者用合法 CA 簽出的假憑證，也因為跟 App 內建的不一致而被拒絕。代價是 certificate 更新或 key rotation 時，如果沒有妥善規劃，可能導致 App 直接無法連線，需要搭配版本更新或多組備援 pin。

題型輪替 #3，第一次出，難度 Easy
Longest Substring Without Repeating Characters（最長不重複子字串）
給定一個字串 s，找出其中不含重複字元的最長子字串長度。 
輸入: s = "abcabcbb"
輸出: 3
解釋: 最長不重複子字串是 "abc"，長度為 3

輸入: s = "bbbbb"
輸出: 1
解釋: 最長不重複子字串是 "b"，長度為 1

輸入: s = "pwwkew"
輸出: 3
解釋: 最長不重複子字串是 "wke"，長度為 3
💡 提示 1：思考方向

最直覺的解法是對每個起點都往右掃到出現重複字元為止，記錄最長長度——但這樣每個起點都要重新掃一次，時間複雜度是 O(n²)。想想看：當你從某個起點往右掃，發現第一次出現重複字元時，下一次真的需要把起點退回「原本起點 +1」，重新從頭掃一次嗎？還是說，你其實已經知道「不含重複字元」的區間上一次延伸到哪裡了，可以直接利用這個資訊？

🔍 提示 2：關鍵觀察

用一個「視窗」（sliding window）維護目前正在檢查、不含重複字元的區間 [left, right]，搭配一個 Set（或 [Character: Int] 記錄字元最後出現的位置）：

  • right 一路往右擴張視窗，每次把新字元加進去
  • 如果新字元在目前視窗內已經出現過，代表視窗不能再涵蓋這個重複的位置——不用把 left 退回起點重新掃，而是直接把 left 跳到「重複字元上次出現位置的下一格」，視窗只往前收縮到剛好排除重複，不多不少
  • 每次視窗合法（無重複）時，用 right - left + 1 更新最長長度

因為 left 和 right 都只會單調往右移動、各自最多走 n 步，整個過程只需要一次線性掃描，時間複雜度從 O(n²) 降到 O(n)。這是 sliding window 題型最典型的套路：用「維護一個滿足條件的區間，隨資料流動態伸縮邊界」取代「每個起點都重新算一次」。

✅ 提示 3：參考解法（Swift） 
func lengthOfLongestSubstring(_ s: String) -> Int {
    let chars = Array(s)
    var lastSeenIndex: [Character: Int] = [:]
    var left = 0
    var maxLength = 0

    for right in 0..<chars.count {
        let c = chars[right]
        if let seenAt = lastSeenIndex[c], seenAt >= left {
            // 重複字元在目前視窗內，收縮 left 到它的下一格
            left = seenAt + 1
        }
        lastSeenIndex[c] = right
        maxLength = max(maxLength, right - left + 1)
    }
    return maxLength
}

複雜度：時間 O(n)（left、right 各自單調遞增，總移動次數不超過 2n），空間 O(min(n, Σ))，Σ 是字元集大小（例如只有小寫英文字母時最多 26）。這裡有個容易忽略的細節：判斷重複時要檢查 seenAt >= left，而不是只檢查「這個字元有沒有出現過」——因為字元可能出現在目前視窗之前（已經被排除在視窗外），那種情況不算重複，不需要收縮 left。如果面試官追問「為什麼不能直接把 left 往前掃到找到重複為止」，可以強調這正是視窗維護「上次已知邊界」的價值——避免每次都要重新掃描。
